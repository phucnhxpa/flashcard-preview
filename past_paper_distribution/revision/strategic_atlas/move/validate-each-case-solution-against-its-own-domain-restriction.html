Algebra & Functions

Validate each case solution against its own domain restriction

Family: Domain Checking Frequency: 1 questions

Trigger — When to use

Fires whenever a problem has been split into cases, each case carrying a domain restriction — typically piecewise functions, modulus equations |f(x)| = g(x), or equations involving √x, log x, or rational expressions with excluded values. Cue phrases include 'for x > a', 'on this interval', or a piecewise definition like f(x) = {... for 0 < x ≤ 3; ... for x > 3}. The move is mandatory once a candidate solution has been produced from a single case: it does not become a valid solution of the original equation until its membership in that case's domain is confirmed.

Motivation — Why this works

Within a case, you replace f(x) by a specific local formula that only equals f(x) on that case's interval. Solving the resulting equation may yield x-values where the two underlying expressions intersect, but if x lies outside the case's interval, the substitution was never valid there, so the solution is extraneous. Domain validation enforces the logical conditional 'IF x is in this interval THEN f(x) = (this formula)' — without the check you have only solved a related, unrestricted equation. This is why mark schemes treat the rejection/acceptance step as a scoring point.

Thought Process — The step-by-step reasoning

  1. Notices that the equation just solved was the case-specific equation, not the original — so each root is only a candidate.
  2. Rules out treating both algebraic roots as automatically valid; recalls that a root from Case 1 must satisfy Case 1's domain, not Case 2's.
  3. Commits to filtering: writes each candidate next to its parent interval and asks 'is this x in that interval?'
  4. Bridges to operation by listing candidates with their domains: (x = −38/5, need 0 < x ≤ 3) and (x = 62/3, need x > 3).
  5. Tactically performs a back-substitution into the original f(x) and the other side g(x) to confirm equality, guarding against arithmetic slips made during the case-solving.

Mechanic — Concrete operations

  1. List each candidate solution alongside the domain of the case that produced it.
  2. Test membership: substitute the numerical value into the domain inequality.
  3. Reject any candidate failing its own domain; mark as extraneous.
  4. Accept candidates that satisfy their domain; collect surviving roots as the solution set.
  5. (Optional belt-and-braces) Substitute surviving roots into the original equation to verify both sides agree.

Worked Micro-Example

Solve $|2x-1| = \tfrac12 x + 30$. Case 1 ($x < \tfrac12$): $1-2x = \tfrac12 x + 30 \Rightarrow x = -\tfrac{58}{5} = -11.6$. Check: $-11.6 < \tfrac12$ ✓ accept. Case 2 ($x \ge \tfrac12$): $2x-1 = \tfrac12 x + 30 \Rightarrow x = \tfrac{62}{3} \approx 20.67$. Check: $20.67 \ge \tfrac12$ ✓ accept. Both survive; if either had failed its inequality it would be discarded.

Variants — Same move, different surface

Modulus equation $|f(x)| = g(x)$ split into $f(x) \ge 0$ and $f(x) < 0$ branches
Validate each root against the sign condition on $f(x)$, not against an x-interval directly.
Piecewise function with three or more pieces
Same procedure applied per piece; ensure boundary points are assigned to exactly one piece as defined.
Logarithmic equations after exponentiating, e.g. $\ln(x-2)+\ln(x+1)=\ln 4$
Reject roots making any argument ≤ 0; the 'domain' is the natural domain of the logs rather than a case interval.
Equations involving $\sqrt{f(x)}$ after squaring
Squaring can introduce extraneous roots; check each candidate satisfies $f(x) \ge 0$ AND the original (unsquared) equation.
Rational equations after clearing denominators
Reject roots that make any original denominator zero.
Trigonometric equations on a restricted range, e.g. $0 \le \theta < 2\pi$
Filter the general solution down to those values lying inside the stated range.

Common Misreadings

  • Thinking that any algebraic solution of the case equation automatically solves the original equation.
  • Confusing the candidate's case label and checking it against the wrong interval.
  • Believing that verification by substitution can replace the domain check (a root may satisfy the case formula = g(x) numerically yet still lie in the wrong region for f).

Pitfalls

  • Skipping the rejection step entirely and listing both roots as final answers — the most common scheme-penalised error in modulus questions.
  • Mishandling boundary points (≤ vs <) so a root sitting exactly on a case boundary is double-counted or wrongly rejected.
  • Arithmetic slips when re-checking inequalities involving negative fractions, e.g. comparing $-38/5$ with $0$.

A-level Scope Notes

Standard across Edexcel IAL P2/P3 modulus and piecewise problems and Further Pure work involving conditional cases; uses only inequality-checking and substitution, no tools beyond the specification.
