Algebra & Functions

Bisect the bracketing interval and retain the sign-change half to localise a root

Family: Interval Bisection Frequency: 3 questions

Trigger — When to use

Fires when a question explicitly says 'use interval bisection' (sometimes 'apply interval bisection n times') starting from a bracket [a,b] on which f has already been shown to change sign. Equivalent phrasings include 'show that α lies in an interval of width 0.1' (where the required number of halvings is dictated by the width reduction (b−a)/2^n), or 'continue bisection until the interval has width …'. The cue is always a continuous f, a known sign-change interval, and a request to systematically narrow it. If the question only asks to 'show a root lies in [a,b]', that is the precursor sign-change move (IVT), not bisection itself.

Motivation — Why this works

Interval bisection exploits the Intermediate Value Theorem: if a continuous f changes sign on [a,b], it changes sign on at least one of [a, m] or [m, b] where m = (a+b)/2. Evaluating f(m) and comparing its sign with f(a) and f(b) tells us which half contains the root, and we discard the other. Each application halves the interval width, so n bisections shrink the bracket by a factor 2^n — guaranteed convergence with no derivative information needed. It is the simplest and most robust numerical root-finder in the FP1/Core Pure toolkit.

Thought Process — The step-by-step reasoning

  1. Notice the question explicitly demands bisection (or specifies a target width, from which the number of halvings is forced, e.g. 0.4 → 0.2 → 0.1 means two bisections).
  2. Rule out alternatives: this is not linear interpolation (which uses a weighted midpoint), not Newton–Raphson (which needs f′), and not algebraic factorisation (the equation typically has no rational root).
  3. Commit to bisection because (i) the bracket [a,b] with f(a)·f(b) < 0 is already in hand from a previous part, and (ii) the method is mechanical — no tactical choice of midpoint is involved.
  4. Bridge to action: compute m = (a+b)/2, evaluate f(m) (in radians for trig functions), and compare its sign to f(a) and f(b).
  5. Tactical decision inside the move: carry forward the previously established sign of f(a) and f(b) rather than recomputing — this saves time and avoids arithmetic slips.
  6. After each step, write the new bracket explicitly and state the sign comparison so the examiner sees the IVT logic; repeat until the required number of bisections (or target width) is reached.

Mechanic — Concrete operations

  1. Identify the current bracket [a, b] with f(a) and f(b) of opposite signs (from a previous part).
  2. Compute the midpoint m = (a + b)/2.
  3. Evaluate f(m) — using radians for trigonometric arguments — and record its sign.
  4. If sign(f(m)) = sign(f(a)), the root lies in [m, b]; if sign(f(m)) = sign(f(b)), the root lies in [a, m]. Discard the other half.
  5. Replace [a, b] by the retained half and repeat for the required number of iterations.
  6. State the final interval and (if asked) note its width = (b−a)/2^n.

Worked Micro-Example

Given f(x) = 2x − 2\sin x − 1 with f(0.2) < 0 and f(0.6) > 0, bisect twice. Step 1: m_1 = 0.4, f(0.4) ≈ −0.179 < 0, so root ∈ [0.4, 0.6]. Step 2: m_2 = 0.5, f(0.5) ≈ +0.041 > 0, so root ∈ [0.4, 0.5]. Hence α ∈ [0.4, 0.5], an interval of width 0.1.

Variants — Same move, different surface

Question asks for three or more bisections, or for a final decimal estimate
Repeat the halving process the stated number of times; for a final estimate, take the midpoint of the last bracket as the approximation to α.
f involves surds, cube roots, or transcendentals (e^x, ln x, sin x)
Method is unchanged — only the arithmetic at midpoints is messier. Keep extra decimal places to avoid sign errors near the root.
Question first says 'show a root lies in [a,b]' before requesting bisection
Do the IVT sign-change argument (with explicit continuity statement) as a precursor part, then feed the bracket into bisection.

Common Misreadings

  • Confusing bisection with linear interpolation (false position) and using a weighted midpoint instead of the arithmetic mean.
  • Performing Newton–Raphson iterations when bisection is explicitly requested.
  • Evaluating f at the midpoint and stopping there, without comparing signs to determine which half to keep.

Pitfalls

  • Choosing the wrong sub-interval by comparing f(m) to the wrong endpoint sign — always pair f(m) with the endpoint of opposite sign.
  • Arithmetic slips at the midpoint, especially with cube roots, surds, or trig functions; these propagate through every later bisection.
  • Using degrees instead of radians when evaluating sin, cos, etc.
  • Stating the new interval without showing the f(m)-value and its sign — examiners require evidence, not just the final bracket.
  • Stopping after the wrong number of iterations: re-check that 2^n matches the requested width reduction.

A-level Scope Notes

This is a standard Edexcel IAL Further Pure (FP1) / Core Pure numerical-methods technique. No formula booklet entry is needed; the IVT is invoked informally via the continuity of polynomials/standard functions. Do not stray into convergence-rate analysis or fixed-point iteration theory — those are out of scope.
