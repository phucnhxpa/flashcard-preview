Trigger — When to use

Fires when prior constraint analysis has reduced the variable(s) to a small explicit finite set (typically 2–6 values), and the claim asks to 'prove ... is always [even/odd/divisible by k/positive/...]'. Cue phrases include 'show that ... for all integers satisfying ...' or 'prove that ... in every case'. Especially natural when an algebraic factor-extraction route looks messy or when the problem already enumerated the admissible values. Edexcel explicitly lists proof by exhaustion as a permitted method.

Motivation — Why this works

Exhaustion is logically complete: if the universe of cases is finite and every case is verified, the universal claim is proven. With only a handful of integer cases, direct computation removes the need for algebraic identities and avoids manipulation errors. The principle is just the definition of 'for all' restricted to a finite domain — checking each member is equivalent to proving the universal statement. Examiners award full marks for a clean, complete exhaustion.

Thought Process — Step-by-step reasoning

  1. Notice that prior constraints have pinned the variable to a small explicit list (e.g. b ∈ {1, 2, 3}).
  2. Rule out a slick algebraic route if it requires more invention than just listing — exhaustion is safer when the case count is tiny.
  3. Commit to exhaustion because it is on-spec, transparent, and almost mark-scheme-proof when executed carefully.
  4. Plan the table: for each value of the controlling variable, compute the dependent quantities and the target expression.
  5. Decide on a closing sentence that explicitly states all cases are covered, so the logical step from 'each case' to 'all cases' is visible to the examiner.

Mechanic — Concrete operations

  1. State the finite case list and justify briefly that these are all the admissible values.
  2. For each case, substitute and compute the target quantity explicitly.
  3. Verify the required property (even, divisible by k, etc.) in each case with a tick or short remark.
  4. Write a concluding sentence: 'In all possible cases the property holds, hence the result is proved.'

Worked Micro-Example

Given $a + b = 7$, $c = b + 2$, with $a, b, c$ positive integers and $a$ even, the constraint forces $b \in \{1, 3, 5\}$... suppose instead the constraint gives $b \in \{1,2,3\}$ with $a = 8 - 2b$, $c = b + 2$. Then: $b=1: abc = 6\cdot1\cdot3 = 18$; $b=2: abc = 4\cdot2\cdot4 = 32$; $b=3: abc = 2\cdot3\cdot5 = 30$. Each is even, so $abc$ is even in all cases. $\blacksquare$

Variants — Same move, different surface

Prove 'always divisible by k' for finitely many integer cases
Compute the quantity for each case and verify divisibility by k explicitly (e.g. show each value is a multiple of 6).
Modular arithmetic claim of the form 'for all integers n, P(n) holds'
Exhaust over residue classes n ≡ 0, 1, ..., k−1 (mod k) instead of over individual integers — finite case set guaranteed.
Prove a parity result for $n^2 + n$ or similar
Split into the two cases n even / n odd and verify each, rather than enumerating numerical values.
Proof in Further Pure where a quadratic Diophantine constraint yields a few integer solutions
List all integer solutions of the constraint, then verify the target property at each solution.

Common Misreadings

  • Checking one or two values and claiming the pattern continues — that is verification by example, not exhaustion.
  • Treating exhaustion as 'giving examples' rather than 'covering every possible case'.
  • Believing exhaustion is informal or low-status; in fact it is an accepted A-level method when the case set is genuinely finite.

Pitfalls

  • Arithmetic slips when computing the target quantity in one of the cases — a single wrong product breaks the proof.
  • Failing to first establish that the listed cases are the only admissible ones.
  • Omitting the concluding 'in all cases' sentence, leaving the universal step implicit.
  • Listing the wrong case set (e.g. forgetting a boundary value or including an inadmissible one).

A-level Scope Notes

Proof by exhaustion is an explicitly named method in the Edexcel IAL Pure specification and is examined in 'prove that ...' questions where constraints reduce the problem to a small finite set of integer cases.
