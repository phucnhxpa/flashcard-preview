Algebra & Functions

Reject extraneous roots by enforcing logarithm domain on original equation

Family: Domain Validation Frequency: 2 questions

Trigger — When to use

After solving a logarithmic equation, you have arrived at an algebraic equation (often linear or quadratic) and obtained candidate values for the variable. The original equation contained terms like log_a(x), log_a(f(x)), or ln(g(x)) whose arguments must be strictly positive. The algebraic step (combining logs, equating arguments, exponentiating) silently dropped the domain restriction, so candidates may include values that make a log argument zero or negative. Phrased differently: 'I have two roots from a quadratic but the original problem involved logs — which roots are actually valid?'

Motivation — Why this works

Logarithms log_a(y) are defined only for y > 0, so any candidate solution that produces a non-positive argument in the ORIGINAL equation is extraneous and must be rejected. The algebraic manipulations (combining log_a(M) + log_a(N) = log_a(MN), or removing logs from both sides) preserve solutions but also introduce spurious ones because they lose the y > 0 information. Substituting candidates back into the original logarithmic statement — not the derived polynomial — is the only reliable filter. Edexcel mark schemes explicitly award a mark for stating which root is rejected and why, so the check is both mathematically and procedurally essential.

Thought Process — The step-by-step reasoning

  1. I notice I solved a logarithmic equation by reducing it to a polynomial (e.g. quadratic), and I now have multiple candidate values.
  2. I rule out simply listing both roots — I recall that taking 'log_a(P) = log_a(Q) ⟹ P = Q' is only valid when P, Q > 0, so my polynomial may admit roots that violate that.
  3. I commit to checking each candidate against the ORIGINAL equation's log arguments (not the derived polynomial, which they all satisfy by construction).
  4. For each candidate, I list every log argument in the original equation and substitute. If any becomes ≤ 0, that candidate is extraneous.
  5. Tactical decision: I check the binding constraint first — usually the term log_a(x) (linear in x) gives a tighter restriction than log_a(x²+c) (always positive). I also remember to check ALL variables involved (e.g. both a and b in a simultaneous system), not only the one I solved for last.
  6. I write a single sentence justifying the rejection (e.g. 'reject b = 0 since log_3(0) is undefined') so the examiner sees the domain check explicitly.

Mechanic — Concrete operations

  1. List the candidate solutions from the algebraic step.
  2. Identify every log (or ln) argument that appears in the ORIGINAL equation: e.g. b, 5b², a, x − 2, etc.
  3. For each candidate, substitute and check whether every argument is strictly positive.
  4. If any argument is ≤ 0, write 'reject [value] since log_a(...) is undefined'.
  5. Retain only candidates passing all checks; state these as the final answer(s).

Worked Micro-Example

Solve $\log_3(b) + \log_3(5b^2) = \log_3(81)$. Combining gives $5b^3 = 81$ on one branch, but a fuller manipulation produces candidates $b = 0$ and $b = 81/5$. Domain check: $\log_3(b)$ requires $b>0$, so $b=0$ is rejected (log of zero undefined); $b = 81/5 > 0$ gives $\log_3(81/5)$ and $\log_3(5\cdot(81/5)^2)$ both defined. Final answer: $b = 81/5$.

Variants — Same move, different surface

Quadratic in a log equation yields one negative and one positive root
Reject the negative root since log_a(negative) is undefined; verify the positive root keeps every log argument positive.
Simultaneous log equations in two variables (a and b)
Check positivity of BOTH a and b in the original equations, not just the variable solved for last.
Equation containing √(expression)
Same domain-validation principle: require expression ≥ 0 for each candidate.
Equation with ln(x − k)
Require x > k for each candidate; reject any root with x ≤ k.

Common Misreadings

  • Checking candidates against the derived polynomial (which they trivially satisfy) instead of the original logarithmic equation.
  • Forgetting that log_a(x²+c) with c > 0 is always defined, and over-zealously rejecting negative roots when the actual constraint comes from a different term.
  • In two-variable systems, verifying only the variable just solved for and ignoring positivity of the other.

Pitfalls

  • Accepting b = 0 because some terms like 5b² evaluate to 0 'without error', overlooking that log_a(0) is undefined.
  • Misestimating surds (e.g. assuming −4 + 4√5 is negative); always compare against √5 ≈ 2.236 carefully.
  • Failing to write the rejection statement explicitly — examiners require the justification, not just the omission of the bad root.

A-level Scope Notes

Within Edexcel IAL P2/P3 logarithm work. Uses only the basic domain fact log_a(y) defined ⟺ y > 0 (no formula booklet entry needed). Do NOT extend to complex logarithms or principal-branch arguments — those are out of scope.
